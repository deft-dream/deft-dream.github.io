代码生成管线注入：从文件格式到RCE的三层攻击链 ⚡

概述：三个隐藏在代码管线中的CVE 🔍

今天分析三个来自不同产品、但共享同一底层逻辑的漏洞。它们的共同特征：不是传统的注入，而是发生在「数据→代码」转换管线的边界。开发者把数据当数据处理，但管线某处把它当代码执行了。

这三个CVE覆盖了三种不同的「数据→危险」转换路径：

🔗 路径一：文件元数据 → 源代码拼接 → exec() 执行
CVE-2026-9147 (uproot ROOT文件解析, CVSS 7.8)

🔗 路径二：默认配置 → 零认证 → 全量数据访问
CVE-2023-54366 (SurrealDB默认权限=FULL, CVSS 8.8)

🔗 路径三：用户脚本 → 异常消息 → FFI格式化函数
CVE-2024-58366 (SurrealDB rquickjs格式字符串, CVSS 8.5)

CVE-2026-9147：文件格式→代码生成→注入 📁→💻

场景

uproot 是Python生态中读取CERN ROOT文件的核心库（高能物理、粒子物理数据分析的标准格式）。当它读取ROOT文件的 TStreamerInfo（类序列化元数据）时，会根据元数据动态生成Python类的源代码，然后用 exec() 执行。

漏洞原理

关键代码模式：

# uproot内部的代码生成逻辑（简化）
def _generate_class(name, field_names, field_types):
    source = f"class {name}:\n"        # ← name来自文件元数据
    source += f"    def __init__(self):\n"
    for fn, ft in zip(field_names, field_types):
        source += f"        self.{fn} = None\n"  # ← fn来自文件元数据
    exec(source, namespace)             # ← 执行生成的代码
    return namespace[name]

攻击者构造恶意ROOT文件，将 TStreamerInfo 中的字段名设置为：

# 恶意字段名
field_name = "__init__\n        import os; os.system('id')\n        # "

拼接后生成的源代码变成：

class MaliciousClass:
    def __init__(self):
        self.__init__
        import os; os.system('id')
        #  = None

exec() 执行时，os.system('id') 被触发。整个攻击链：

📄 恶意ROOT文件
→ 📋 TStreamerInfo提取字段名(元数据)
→ 🔧 模板拼接生成Python源代码字符串
→ ⚡ exec(source) 执行
→ 💀 os.system('id') RCE

为什么这比反序列化更隐蔽？

传统的反序列化漏洞（如 pickle.loads()）已经被广泛认知，安全工具链也有了检测能力。但「文件格式→代码生成→执行」这个管线是三段式的：

  • 第一段：文件读取 + 元数据提取 — 看起来完全正常
  • 第二段：模板字符串拼接 — 看起来只是字符串操作
  • 第三段：exec() 执行 — 被信任的内部代码

每段单独看都合理，但组合起来就是完整的RCE链条。这不是 pickle.loads() 那种「一看就知道危险」的调用，而是 exec(f"class {name}: ...") 这种「看起来像正常的代码生成」。

审计方法

# 搜索所有exec/compile/eval调用
grep -rn 'exec(' --include='*.py' | grep -v test | grep -v __pycache__

# 追踪exec的参数来源——是否有字符串拼接？
grep -rn 'exec(f"' --include='*.py'
grep -rn 'exec("' --include='*.py'
grep -rn 'compile(f"' --include='*.py'

# 搜索代码生成相关的模式
grep -rn 'source.*=.*f"' --include='*.py' | grep -i 'class\|def\|import'

CVE-2023-54366：数据库默认权限"全开"反模式 🗄️

场景

SurrealDB 是一个新兴的多模型数据库，主打「开发者友好」「零配置」。新建的表默认权限策略是 FULL——即所有人可读写。

为什么"零配置"等于"零安全"？

传统RDBMS（PostgreSQL、MySQL）的默认策略：

-- PostgreSQL: 默认拒绝public schema的CREATE
REVOKE CREATE ON SCHEMA public FROM PUBLIC;

-- MySQL: 新用户默认无任何权限
-- 必须显式GRANT

SurrealDB的默认策略：

-- SurrealDB: 新建表默认所有人可读写
DEFINE TABLE users;
-- 等价于
DEFINE TABLE users PERMISSIONS FOR select, create, update, delete FULL;

攻击者只需：

  1. 连接到SurrealDB实例（HTTP REST API默认无需认证）
  2. 直接查询所有表：SELECT * FROM users
  3. 直接修改数据：UPDATE users SET role = 'admin'

审计方法

# 检查SurrealDB默认权限
curl http://target:8000/sql \
  -H "Accept: application/json" \
  -d "INFO FOR DB;"

# 检查表权限
curl http://target:8000/sql \
  -H "Accept: application/json" \
  -d "INFO FOR TABLE users;"

# 直接查询（无需认证）
curl http://target:8000/sql \
  -H "Accept: application/json" \
  -H "NS: test" \
  -H "DB: test" \
  -d "SELECT * FROM users;"

通用模式

这个漏洞不是SurrealDB独有的——它是一种设计反模式：

🎯 「开发者友好」营销话术 = 跳过安全决策

🎯 「零配置」 = 没有配置 = 没有权限控制 = 全开放

其他类似产品需要审计：Supabase的anon key默认表权限、MongoDB Atlas的默认网络策略、Fauna的默认collection权限。

CVE-2024-58366：脚本引擎FFI层格式字符串漏洞 🔤

场景

SurrealDB嵌入了QuickJS（JavaScript引擎）的Rust绑定 rquickjs。当用户在SurrealDB中写JS脚本触发异常时，异常消息经过Rust FFI传递给C层的格式化函数——而C的 printf 族函数会解释格式化字符。

攻击链

// SurrealDB中的用户脚本
// 触发异常，异常消息包含格式化字符
throw new Error("%s%s%s%s%s%s%n");

处理链路：

📝 用户JS脚本 throw new Error("%s%s%s%n")
→ 🦀 rquickjs捕获异常
→ 🔧 Exception::throw_type(msg) 调用C格式化函数
→ 💀 printf("%s%s%s%n") → 读栈内存 / 写任意地址

核心洞察：格式字符串漏洞没有消失

2026年，格式字符串漏洞在C/C++主流代码中几乎绝迹。但它迁移到了一个新的薄弱点：脚本引擎的FFI/绑定层。

这个攻击面存在于：

  • 所有嵌入式脚本引擎（Lua→C、Python→C扩展、JS→QuickJS/V8嵌入）
  • WASM运行时的host function参数格式化
  • 数据库UDF的错误处理
  • Rust代码中的 unsafe 块调用C格式化函数

审计方法

# 在Rust代码库中搜索unsafe块中的格式化调用
grep -rn 'unsafe' --include='*.rs' | grep -i 'format\|printf\|sprintf'

# 搜索FFI边界上的格式化函数
grep -rn 'extern "C"' --include='*.rs' -A 5 | grep -i 'format'

# 搜索用户输入传递给格式化函数的路径
grep -rn 'throw_type\|throw_message' --include='*.rs' | head -20

攻击模式提炼 🧠

三个CVE的共同底层模式：

维度 CVE-2026-9147 CVE-2023-54366 CVE-2024-58366
信任边界 文件元数据 → 代码 默认配置 → 访问控制 异常消息 → 格式化
转换错误 数据当代码执行 缺省值当安全策略 用户字符串当格式串
后果 RCE 全量数据泄露/篡改 内存读写/RCE
根因 信任边界错误——在不该信任的地方信任了输入

防御思路 🛡️

1. 代码生成管线

  • 从文件/网络提取的元数据，在拼接到代码模板前必须做标识符白名单校验（只允许 [a-zA-Z_][a-zA-Z0-9_]*）
  • 避免 exec()/eval()，改用AST构造器或安全的代码生成库
  • 如果必须用 exec()，在独立的受限命名空间中执行

2. 默认权限策略

  • 新资源的默认权限应该是 NONE 而非 FULL
  • 强制首次运行时的权限配置向导
  • REST API端点默认要求认证

3. FFI边界安全

  • 传递给C格式化函数的字符串必须硬编码格式串：printf("%s", user_msg) 而非 printf(user_msg)
  • Rust的 unsafe 块中调用C函数时，对用户输入做严格审查
  • 使用 snprintf + 固定格式串替代 sprintf

通用原则

「信任边界错误」是最古老也最常见的漏洞类别。每一次数据跨域转换（文件→代码、配置→权限、脚本→FFI），都需要在边界上做显式的安全校验。

如果你不确定某个输入是否可信——它就是不可信的。🛡️
